# 🧠 PROJECT CONTEXT — ENTERPRISE E-COMMERCE SAAS PLATFORM

## 🎯 Vision

We are building a **multi-tenant SaaS e-commerce platform** that:

- Competes with Amazon Seller Central, Shopify, WooCommerce
- Supports ALL store types:
  - physical products
  - grocery
  - electronics
  - clothing
  - restaurant/menu
  - services
  - subscriptions
  - digital products (courses, downloads)

BUT with a key difference:

👉 It must be **simpler, faster, and less frustrating to use**

---

## 💡 Core Philosophy

Most platforms:
- are powerful
- but complex
- confusing for merchants
- hard to manage variants, inventory, imports

### Our platform must:

✅ simplify complexity  
✅ reduce merchant effort  
✅ make bulk operations easy  
✅ make imports understandable  
✅ make data visible and usable  
✅ provide real-time clarity (inventory, orders, shipments)

---

## 🧱 System Architecture Principles

### 1. Multi-Store SaaS (STRICT)

- Every entity must be store-scoped
- No cross-store data leaks
- Use `current.store` middleware everywhere

---

### 2. Flexible Product System (CRITICAL)

Products must support:

- simple products
- variant products
- service products
- digital products
- future extensibility

---

### 3. Hybrid Data Model

We DO NOT rely only on strict DB schema.

We use:

| Data Type | Storage |
|----------|--------|
| Core product fields | products table |
| Variants | product_variants |
| Images | product_images |
| Custom mapped fields | products.meta.custom_fields |
| Unmapped import data | products.meta.import_extra |

👉 This allows:
- importing ANY supplier catalog
- supporting ANY store type
- future extensibility

---

### 4. Import System Philosophy (ENTERPRISE)

Import is NOT just CSV upload.

It must:

- accept messy real-world data
- support large catalogs (10k–100k+)
- allow mapping + custom fields
- preserve unknown data safely
- be resumable
- provide clear feedback

---

### 5. Product Usability Layer (NEW FOCUS)

Previously:
- system stored data correctly

Now:
- system must **EXPOSE data correctly**

Product detail must:

- show all relevant data
- differentiate:
  - core data
  - custom mapped data
  - raw imported data
- be readable by non-technical users

---

### 6. Variant Complexity Handling

Variants must support:

- multiple options (size, color, etc.)
- combinations
- variant-level pricing
- variant-level inventory
- variant-level custom fields

And still be:

👉 easy to manage for merchants

---

### 7. Performance & Scale

System must handle:

- 50k+ products
- bulk imports
- async processing
- background jobs
- chunked processing

---

### 8. Merchant Experience (MOST IMPORTANT)

Every feature must answer:

👉 “Does this make merchant life easier?”

If not → it is wrong implementation.

---
